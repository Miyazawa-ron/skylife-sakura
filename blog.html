When You Have Your First AI Employee, How Do You Manage It? 最初のAI社員ができたら、どう管理するのか？
April 18, 2026 · 12 min read 2026年4月18日 · 読了12分
↓

Most people feel surprised the first time they use an AI Agent, confused the second time, and chaotic after the third.

The surprise comes from efficiency. The confusion comes from it sometimes working, sometimes not. The chaos comes from running three or four AI tools at the same time — each operating independently, unaware of each other, with no coordination. The entire workflow becomes harder to maintain than before.

This is not a problem with AI itself. It is a governance problem.

Recently I studied a GitHub repository: Hermes Agent Control Room. It is not an AI application itself, nor a new model, but a methodology and documentation system for managing and orchestrating teams of AI Agents. After reading it, I realized this approach deserves a serious article.

Where Does the Problem Lie?

Let me pour some cold water: most people who claim to be "working with AI" are essentially just chatting with AI.

The difference between chatting and working is that work requires memory, role division, process衔接, and traceable records when problems arise. When every AI conversation starts from zero, when different Agents don't know each other, and when there is no documentation recording "what this AI is responsible for", what you have is not a team — but a group of forgetful temporary workers.

"Build the Control Room first, then plug in Agents. Avoid turning AI into a bunch of scattered robots."

This sentence seems simple, but the logic behind it is worth pondering: it analogizes the management of AI Agents to human resource management in a company. You wouldn't hire a bunch of employees and let them coordinate themselves. You need job responsibilities, reporting chains, and task distribution systems.

What is Control Room?

Control Room is not where AI runs, but where AI is described, regulated, and governed. In technical terms, it is a "control plane" separated from the actual running Agent instances.

For example: Control Room is like a company's HR system and internal Wiki — it records who everyone is, what they are responsible for, what rules exist, where passwords are stored, and who to contact when problems arise. But it doesn't do the work itself. The Agents below do the actual work.

Four Growth Levels

One of the most valuable parts of this system is that it defines a growth path from "personal toy" to "team infrastructure", divided into four levels, each with clear applicable scenarios.

For myself, I am currently around Level 1.5 — multiple tools are running, but roles are vague and there is no documentation. This grading made me realize what I should do next.

Task Bus: Enabling Agents to "Talk" to Each Other

The core problem of multi-Agent collaboration is communication: when one Agent completes a task, how does the result get passed to the next?

The solution in this repository is very engineering-oriented and simple — using file directories to simulate a message queue. Each Agent has four folders: inbox, working, outbox, and archive.

The elegance of this design lies in the fact that it does not rely on any middleware or third-party services. The file system itself becomes the message queue. It is simple, auditable, and when problems occur, you can open the folder to see the status.

Security: The Easiest Rule to Break

In multi-Agent scenarios, security is a real issue, not a theoretical risk. When you have four or five Agents sharing API Keys, and keys are scattered across configuration files, the probability of leakage or misuse rises rapidly.

This repository proposes a separation principle: Record the name and location of keys, never record the value of the keys.

The value of this rule is that the Control Room documentation can be discussed publicly, uploaded to Git, and collaborated on by multiple people — because it only contains structure, not secrets.

8 Built-in Skills, Ready to Use

In addition to the methodology, this repository also includes 8 reusable "Skills" — each Skill is a standard operating manual that tells Agents how to complete specific tasks.

My Inspiration and Calm Assessment

After reading this repository, my first reaction was excitement, and my second reaction was restraint.

The reason for excitement: this approach helped me clearly see the root of chaos in my own workflow — I had tools, but no governance. Each AI tool operated independently, with no role definitions, no documentation, and no unified task tracking.

The reason for restraint: this repository currently has only 7 commits. There is a clear gap between the 248 stars and the actual completion level of the repository. It is more like a carefully designed blueprint than a system that has been validated in large-scale production. The entire architecture assumes you are running Ubuntu/Debian VPS, and is not directly applicable to macOS local environments.

大多数人第一次使用AI Agent时的感受は惊喜，第二次は困惑，第三次以后则是混乱。

惊喜来自于效率。困惑来自于它有时灵、有时不灵。混乱来自于——当你同时运行着三四个AI工具时，它们各自为政、互不知晓、没有协调，整个工作流开始变得比之前更难维护。

这不是AI本身的问题。这是一个治理问题。

最近我在研究一个GitHub仓库：Hermes Agent Control Room。它不是AI应用本身，也不是什么新模型，而是一套用来管理和编排AI Agent团队的方法论与文档体系。看完之后我意识到，这套思路值得认真写一篇文章。

问题出在哪里？

先泼一盆冷水：大多数声称在"用AI工作"的人，本质上是在用AI聊天。

聊天和工作的区别在于——工作需要记忆、角色分工、流程衔接，以及出了问题时能追溯的记录。当你的AI对话每次都从零开始，当不同Agent之间互不相识，当没有任何文档记录"这个AI负责什么"，你拥有的不是团队，而是一群健忘的临时工。

"先搭控制室，再插Agent。避免把AI跑成一堆零散的机器人。"

这句话看起来简单，但背后的逻辑却值得细品：它把AI Agent的管理问题，类比成了企业的人力资源问题。你不会直接招一批员工然后让他们自己协调，你需要岗位职责、汇报链条、任务分发系统。

Control Room是什么？

Control Room不是运行AI的地方，而是描述、规范和治理AI的地方。用技术语言说，这是一个"控制平面"，与实际运行的Agent实例分离。

打个比方：Control Room类似一家公司的HR系统和内部Wiki——它记录每个人是谁、负责什么、有哪些规则、密码放在哪、出了问题找谁。但它本身不干活，干活的是下面的Agent。

四个成长级别

这套体系最有价值的部分之一，是它定义了一条从"个人玩具"到"团队基础设施"的成长路径，分为四个级别，每一级都有清晰的适用场景。

对我自己而言，目前大概是Level 1.5——有多个工具在跑，但角色模糊、没有文档。这套分级让我意识到下一步应该做什么。

Task Bus：让Agent之间能"传话"

多Agent协作的核心问题是通讯：一个Agent完成了任务，结果怎么传给下一个？

这个仓库的解法很工程化，也很朴素——用文件目录模拟消息队列。每个Agent有自己的收件箱（inbox）、处理中（working）、发件箱（outbox）和归档（archive）四个文件夹。

这个设计的优雅之处在于：它不依赖任何中间件或第三方服务，文件系统本身就是消息队列。简单、可审计、出了问题打开文件夹就能看到状态。

安全问题：一条最容易犯错的铁律

在多Agent场景下，安全是个真实的问题，不是理论上的风险。当你有四五个Agent共享API Key，当密钥散落在各处的配置文件里，泄露或误用的概率会快速上升。

这个仓库提出了一条分离原则：记录密钥的名称和位置，绝不记录密钥的值。

这条规则的价值在于：Control Room的文档可以公开讨论、可以上传Git、可以多人协作——因为里面只有结构，没有秘密。

8个内置技能，拿来就能用

除了方法论，这个仓库还内置了8个可复用的"Skill"——每个Skill是一套标准操作手册，告诉Agent如何完成特定任务。

对我自己的启发与冷静评估

看完这个仓库，我的第一反应是兴奋，第二反应是克制。

兴奋的原因是：这套思路帮我看清楚了我自己工作流的混乱根源——工具有，但治理没有。每个AI工具各自为政，没有角色定义，没有文档，没有统一的任务追踪。

克制的原因是：这个仓库目前只有7次提交，248星的热度和仓库本身的完成度之间有明显落差。它更像一份精心设计的蓝图，而不是一套经过大量生产验证的系统。整个架构假设你在跑Ubuntu/Debian VPS，对macOS本地环境不直接适用。
